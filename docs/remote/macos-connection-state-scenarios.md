# macOS Remote Connection State Scenarios

This matrix is the source of truth for macOS Remote connection-state behavior. The client cannot directly know the host power state; it infers the UI state from observable signals and preserves the last usable snapshot whenever the host is not reachable.

| Scenario | Observable signals | Inferred condition | Expected host state | Polling cadence | Cache / standalone behavior | Remote command enablement | Recovery trigger | Test coverage |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| Normal online foreground polling | Tailscale probe skipped or reachable, `/remote/status` succeeds | Host and Remote Agent are usable | `online` / `페어링됨` | User base interval, default 5s | Mirror latest payload when `state_revision` changes; otherwise refresh local progress only | Launch and remote power allowed by host capabilities | Next status success with changed revision refreshes payload | ViewModel smoke, static tests |
| Stable online background/idle | App inactive or repeated unchanged `state_revision` | Host is usable but no visible update is needed | `online` / `페어링됨` | Back off toward 30-60s | Keep last payload and local progress current without full sync churn | Commands remain enabled when app is active again | User action, app activation, changed revision, or immediate command probe | Static tests |
| External host shutdown / hibernate | Tailscale host probe returns no reply / timeout | Host is off, asleep, hibernating, or Tailscale is down | `offlineExpected` / `호스트 꺼짐` | Short recovery schedule, then 60s | Preserve cached game cards and recompute local progress | Remote launch disabled; local wake allowed if configured | Tailscale reachable plus status success | Supervisor smoke, ViewModel smoke |
| External host boot / reboot while client was offline | Previous state was disconnected, then `/remote/status` succeeds | Host recovered independently of client action | `online` / `페어링됨` | Reset to user preference | Force payload sync even if `state_revision` is unchanged | Re-enable online commands after sync | Status success | Supervisor smoke |
| Tailscale reachable but HTTP server down | Tailscale probe succeeds; HTTP status times out or cannot connect | Host network is reachable, Remote Agent is unavailable | `agentUnavailable` / `서버 응답 없음` | Short reconnect burst, then 60s | Preserve cache and local progress | Remote launch/power disabled except local wake policy | HTTP status success | Supervisor smoke |
| Slow HTTP status success | `/remote/status` succeeds but diagnostics show repeated slow responses | Host is usable but should not be hammered | Keep current state, usually `online` | Back off after repeated slow status responses | Preserve normal UI and avoid full sync unless revision/recovery requires it | Commands remain based on state, not latency alone | Faster status responses or user command immediate probe | Static tests |
| Tailscale CLI unavailable | Tailscale CLI missing/not running/not logged in | Tailscale signal cannot be trusted | Keep current state until HTTP result | HTTP fallback cadence | Preserve cache on HTTP failure | Based on HTTP result | HTTP status success/failure | Static tests |
| LAN / loopback URL | Host is loopback, LAN, or non-Tailscale URL | Tailscale ping should not be used | Based on HTTP result | HTTP fallback cadence | Preserve cache on HTTP failure | Based on HTTP result | HTTP status success | Static tests |
| Auth rejected | HTTP 401/403 | Stored token is rejected by host | `authRejected` / `인증 확인 필요` | 60s | Preserve token and cache; show pairing guidance | Remote commands disabled | Explicit token refresh or successful status with valid token | Supervisor smoke |
| Client wake command accepted | User clicks wake and local/remote adapter accepts | Host is expected to boot soon | `waking` / `부팅 대기 중` | Immediate probe, then wake schedule: 1s x15, 2s x15, 5s x24 | Preserve cache and local progress | Launch disabled until online; wake button may remain local-only | Status success forces payload sync | Supervisor smoke, ViewModel smoke |
| Client sleep / shutdown accepted | User clicks sleep/shutdown and the power adapter returns an explicit command-accepted signal | Host is expected to disappear | `goingOffline` / `종료 확인 중`, then `offlineExpected` | 2s x5, then 60s | Preserve cache and local progress | Remote commands disabled except local wake | Tailscale no reply or status/power hint confirms offline | Supervisor smoke, ViewModel/static tests |
| Client restart accepted | User clicks restart and adapter accepts | Host is expected to disconnect and return | `restarting` / `재시동 대기 중` | 5s, then wake schedule | Preserve cache and local progress | Remote commands disabled until online | Status success | Supervisor smoke |
| Cached progress while disconnected | Any disconnected state and cached processes exist | Standalone display should remain useful | Current disconnected state | State-specific cadence | Recompute cycle/stamina locally every 30s | Launch disabled until online | Status success and payload sync | ViewModel smoke |
| Recovery with unchanged revision | Previous state was not online; status succeeds with same revision | Host recovered but revision may not reveal missed cache/UI drift | `online` / `페어링됨` | Reset to user preference | Force payload sync once | Commands re-enable after sync | First status success after disconnected state | Supervisor smoke |
| Mac sleep then resume | `NSWorkspace.didWakeNotification` or app activation after disconnected/stale state | Client tasks may have paused and local progress is stale | Keep current state until immediate probe result | Immediate probe, then state-specific cadence | Refresh local progress immediately | Do not change command enablement until probe result | Immediate status/probe result | Supervisor smoke |
| Game launch accepted | User clicks a game launch button and host returns accepted command result | Command was delivered; process/session detection may lag | Remain `online` / `페어링됨` | Immediate command-scoped mirror, then launch chase: ~0.75s, 1.5s, 3s | Refresh process payload quickly; do not run global refresh UI | Only the clicked row enters pending/disabled state | Process becomes running, revision changes, chase expires, or connection fails | ViewModel smoke, static tests |
| Game launch connectivity failure | Launch POST fails due auth/connectivity | Command was not delivered or auth expired | Supervisor-derived state (`reconnecting`, `agentUnavailable`, `offlineExpected`, or `authRejected`) | State-specific reconnect/backoff | Preserve cache; clear row pending state | Commands disabled according to supervisor state | Status success or token recovery | Static tests |

## Task 2 polling contract

- Host push/SSE/WebSocket is intentionally out of scope for this pass.
- Client commands are still asynchronous signals, but the client owns the post-command observation window.
- `/remote/status` remains the hot path. Full payload sync happens on `state_revision` change or online recovery; launch uses process-scoped fast mirroring so the connection pill does not flicker to global `동기화 중`.
